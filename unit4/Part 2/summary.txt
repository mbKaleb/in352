We took DeadLockExample with two synchronized blocks, each with its own lock. The program was designed to halt due to dead lock. After executing the program our terminal process were waiting and never exited. This displayed deadlock. After examining the compilationg with javap -c -p. Java print -c for disassembled bytecode instructions, and -p for private classes. This output showed us how the code is actually executed under the hood by the JVM. 

Each synchronized block comiples to a monitorenter blocking other threads from entering. The monitorexit releases that lock. One thing to note here is that Java guarantees a lock is released even if an exception occurs inside the block, the compiler uses two monitorexit instructions per synchronized block. This is why function 0 and 1 show two nested symbols blocks (lock1/lock2 and lock2/lock1) with four monitorexit.
